## 3. 화이트박스 관점 핵심 분석 결과(초기)

| 구분 | 관찰 내용 | 관련 가이드라인 항목(예비) | 비고 |
|---|---|---|---|
| 인증 비밀키 | `SECRET_KEY` 기본값 존재(`default_secret_key_for_dev`) | IA, IS, CC | 운영환경 강제 설정 필요 |
| CORS 정책 | `allow_origins` 기본값 `*` + `allow_credentials=True` | CF, SN | 환경별 origin 제한 필요 |
| 초기 비밀번호 | 회원가입/권한부여 시 고정 비밀번호 사용 로직 존재 | BF, IA | 강한 비밀번호 정책 필요 |
| 토큰 저장소 | 프론트에서 JWT를 `localStorage`에 저장 | IS, CC, IL | 저장전략/보호정책 검토 필요 |
| 권한검증 | 관리자/슈퍼관리자 dependency 분리 적용 | IN | 세부 엔드포인트별 재검증 예정 |
| 에러처리 | 상세 에러문구 반환 구간 존재 | EP, IL | 오류 응답 표준화 필요 |

> 주의: 본 표는 STEP 2의 예비 분석이며, 최종 판정은 STEP 4 진단 및 STEP 6 재진단 결과로 확정한다.

## 4. 점검 단위 분해(모듈)

### M2-1. 인증/인가 분석
- JWT 발급/검증
- 관리자 권한 검증 흐름
- 인증 우회 가능성 점검 포인트 정의

### M2-2. 세션/토큰/쿠키 분석
- 토큰 저장 위치 및 노출 가능성
- 세션 만료/재발급 정책
- 클라이언트 보관 데이터 최소화 여부

### M2-3. 입력/출력 검증 분석
- API 요청 파라미터 검증
- 출력 인코딩 및 프론트 렌더링 안전성
- 파일 처리 엔드포인트 유무 및 보호수단

### M2-4. 운영 보안설정 분석
- CORS, HTTP Method 허용 범위
- 오류 응답/로그 정책
- 관리자 경로 노출 및 접근 제어

## 6. 판정/위험도 표기 원칙 적용

- 판정값은 `양호` 또는 `취약`만 사용
- 기능/엔드포인트가 부재한 경우는 `N/A`로 별도 표기(예: FU/FD)
- 위험도는 `위험평가 보고서.xlsx` 모델로 산정
- 본 단계에서는 가이드라인 항목 중요도(`상/중/하`) + 위험도 계산 입력값을 함께 관리

### 6.1 위험평가 모델 입력값 정의(STEP2 확정)

| 입력 항목 | 값/기준 | 비고 |
|---|---|---|
| 항목중요도 | WebApp 21항목 전부 `상(3)` | 2026 가이드라인 기준 |
| 위협도 | PARK-01~PARK-21 전부 `3` | 엑셀 `2.위협 목록` 시트 |
| 평가가중치 | 양호=1, 취약=0, N/A=제외 | STEP4 판정 결과 연동 |
| 자산중요도 C/I/A | STEP2 자산분류 기준으로 부여 | 서비스/API/DB 단위 |

### 6.2 자산 중요도(C/I/A) 초안

| 자산 | C | I | A | 근거 |
|---|---:|---:|---:|---|
| parktel-frontend (웹 서비스) | 2 | 3 | 3 | 서비스 연속성/변조 영향 높음 |
| FastAPI `/api/*` | 3 | 3 | 3 | 인증·신청·관리 기능 핵심 API |
| 사용자/신청/공지 DB | 3 | 3 | 2 | 개인정보·업무 데이터 중심 |

> 최종 C/I/A 값은 STEP4 실진단 시 자산 오너와 확정

### 6.3 STEP 3 전달용 중요도 표기 템플릿

| 항목코드 | 항목명 | 중요도(상/중/하) | 근거 |
|---|---|---|---|
| CI | 코드 인젝션 | 상 | 주요정보통신기반시설 가이드라인(WebApp) |
| SI | SQL 인젝션 | 상 | 주요정보통신기반시설 가이드라인(WebApp) |
| XS | XSS | 상 | 주요정보통신기반시설 가이드라인(WebApp) |
| CF | CSRF | 상 | 주요정보통신기반시설 가이드라인(WebApp) |
| IN | 불충분한 권한 검증 | 상 | 주요정보통신기반시설 가이드라인(WebApp) |
