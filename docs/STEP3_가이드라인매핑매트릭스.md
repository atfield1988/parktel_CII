# STEP 3 산출물: 가이드라인 매핑 매트릭스

- 프로젝트명: 주요정보통신기반시설 가이드라인 기반 Web Application 취약점 점검 및 시큐어코딩
- 작성일: 2026-04-27
- 입력 문서
  - `docs/STEP1_착수보고서.md`
  - `docs/STEP2_기존코드분석서.md`
- 목적: 가이드라인 Web Application 21개 항목을 대상 코드/기능/엔드포인트와 1:1 매핑하여 STEP 4 화이트박스 진단 입력자료로 사용

---

## 1. 매핑 원칙

1. 기본 판정값은 `양호` 또는 `취약` 사용(진단 결과 단계)
2. 점검 대상 기능/엔드포인트가 코드상 존재하지 않을 경우 `N/A`로 표기
3. 위험도는 `위험평가 보고서.xlsx` 모델을 적용
4. 2026 상세가이드 기준 Web Application 21개 항목 중요도는 모두 `상`
5. `kisa-webapp-vuln.skills`와 코드명이 일부 상이할 경우, **가이드라인 원문 코드체계(예: AU=자동화 공격)를 기준으로 고정**하고 스킬 항목은 보조 점검으로 병행
6. 스킬 점검항목 중 추가 통제(예: 쿠키속성, 로깅마스킹)는 EP/IL/CC/AU 세부 점검 포인트로 흡수 반영

## 3. 위험평가 모델 매핑(PARK 코드 연동)

| WebApp 코드 | 위협코드 | 위협도 | 적용 방식 |
|---|---|---:|---|
| CI~WM (21개 전체) | PARK-01~PARK-21 | 3 | STEP4에서 판정(양호/취약/N/A) 결과와 결합하여 위험도 산출 |

- 위협코드는 `위험평가 보고서.xlsx` `2.위협 목록` 시트 기준
- 각 항목 위험도 산식은 STEP1 4.2 절을 준수
- N/A 항목(FU/FD)은 위험도 총계에서 제외하고 사유를 별도 기재

---

## 4. FU/FD 항목 존재성 검증 근거

- Backend 전수 검색 결과
  - `UploadFile|File\(|FileResponse|StreamingResponse|multipart` : 매치 0건
  - `upload|download` (routers): 매치 0건
- Frontend 전수 검색 결과
  - `upload|download|FormData|multipart/form-data` : 매치 0건
- 결론
  - 현재 코드베이스 기준 파일 업로드/다운로드 기능이 구현되지 않아 FU/FD는 `N/A` 처리
